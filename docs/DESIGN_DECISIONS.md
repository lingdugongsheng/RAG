# 设计决策记录（Architecture Decision Records）

本文档记录 Advanced RAG System 在架构演进过程中所做的重要技术决策及其理由，便于后续维护与扩展。

---

## 1. 整体架构：从管道到智能体

### 决策
使用 LangGraph 构建 Agentic RAG，而非传统的固定流程管道。

### 理由
- 固定管道（检索→生成）无法应对复杂问题（需多步推理、工具组合）和动态场景。
- LangGraph 提供状态图抽象，支持条件分支、循环、并行，天然适合构建自主决策的 Agent。
- 通过图结构，可独立开发、测试每个节点（规划、工具调用、反思），便于功能扩展和错误隔离。

### 后果
- 增加了系统复杂度，但大幅提升了智能程度和灵活性。
- 状态管理需显式定义，我们使用 TypedDict 明确 AgentState 的结构，保证了可维护性。

---

## 2. 知识图谱引入（Graph RAG）

### 决策
集成 Graph RAG，允许从文档中抽取实体关系并构建知识图谱，用于多跳推理查询。

### 理由
- 纯文本检索在回答实体关系类问题（如“张三的导师是谁？”）时效果不佳。
- 通过 LLM 抽取三元组构建图谱，可捕获文档中的结构化知识，实现图上的多跳遍历。
- 图谱检索与向量、BM25 检索融合后，能显著提升对关系型问题的召回率和答案质量。

### 后果
- 入库时增加了三元组抽取步骤，消耗一次 LLM 调用，但成本可控。
- 图谱目前基于内存 networkx，重启后丢失。后续可迁移至 Neo4j 等持久化图数据库。

---

## 3. 多路混合检索（向量 + BM25 + 图谱）

### 决策
采用三路检索融合：稠密向量检索（ChromaDB）、稀疏关键词检索（BM25）、知识图谱检索。

### 理由
- 向量检索擅长语义匹配，但可能遗漏精确关键词；BM25 擅长关键词匹配，互补性强。
- 图谱检索补充了实体关系的结构化信息，尤其在多跳推理场景中不可替代。
- 通过 RRF 融合，避免了不同检索器分数分布差异带来的归一化问题。

### 后果
- 检索延迟略有增加，但通过并行执行和轻量级图谱（networkx）优化，实际影响小。
- 需维护多套存储，但各组件接口清晰，替换成本低。

---

## 4. RRF（倒数秩融合）算法

### 决策
使用 RRF 融合多路检索结果，而非线性加权或其他复杂融合方法。

### 理由
- RRF 无需对分数进行归一化，对排名变化鲁棒。
- 公式简单（1/(k+rank)），调参友好（仅需 k 值），已在多路检索工业实践中验证有效。
- 我们的实验发现 RRF 能够稳定提升混合检索的 MRR 和召回率。

### 后果
- 融合结果忽略了原始分数的区分度，但在召回阶段以排序为主要目标时足够优秀。

---

## 5. 重排序策略：Embedding 相似度 vs Cross-Encoder

### 决策
当前使用 Embedding 相似度（余弦相似度）对候选文档重排序，而非专门的 Cross-Encoder 模型。

### 理由
- 避免额外下载和部署大模型，降低环境搭建难度和资源消耗。
- 在当前中小规模数据集下，Embedding 相似度已能提供有效的排序提升。
- 架构上预留了替换接口：只需实现一个新的 Reranker 类即可切换为 Cross-Encoder，实现平滑升级。

### 后果
- 排序精度略低于 Cross-Encoder，但在现阶段性价比最高。

---

## 6. 向量数据库选型：ChromaDB

### 决策
选择 ChromaDB 作为默认向量存储。

### 理由
- 轻量级，可直接嵌入 Python 进程，无需独立服务，适合演示和中小规模部署。
- 与 LangChain 深度集成，API 简洁。
- 支持内存模式（开发）和持久化模式（生产），灵活切换。

### 后果
- 水平扩展能力弱于 Milvus/Qdrant，未来如需处理亿级数据需迁移。

---

## 7. 联网搜索工具：DuckDuckGo (ddgs)

### 决策
使用 DuckDuckGo 作为网络搜索后端，并通过 `ddgs` 库（前身为 duckduckgo_search）调用。

### 理由
- 免费、无需 API Key，适合开源项目和演示。
- 搜索质量在日常查询中足够可靠。
- 原库已被重命名为 `ddgs`，我们及时更新以消除弃用警告。

### 后果
- 网络搜索受限于 DuckDuckGo 的可访问性（某些地区可能不稳定），已实现失败回退至本地检索。

---

## 8. Agent 规划原则：优先本地检索

### 决策
在规划 Prompt 中明确要求优先使用本地知识库（local_search 或 graph_search），仅在本地无法满足时才使用联网搜索。

### 理由
- 保护数据隐私，避免敏感信息外传。
- 降低对外部服务的依赖和延迟，提高响应速度。
- 通过反思机制，本地检索不足时还可以重试或触发联网。

### 后果
- 对于实时性问题（如天气），Agent 可能需要多次规划才能正确选择 web_search，但整体可控。

---

## 9. 异常处理：工具重试与回退

### 决策
为每个工具调用实现基于 `_execute_with_retry` 的通用重试机制，并设置回退策略（如网络搜索失败后自动回退至本地检索）。

### 理由
- 外部服务（LLM、搜索引擎、向量数据库）可能出现瞬时错误，自动重试可提升系统鲁棒性。
- 回退策略确保核心问答流程不会因单个工具异常而完全中断。

### 后果
- 增加了调用次数和延迟，但通过可配置的重试次数和延迟时间，可在可靠性间取得平衡。

---

## 10. 并行执行

### 决策
当 Agent 计划生成并行工具组（如同时执行 local_search 和 graph_search）时，使用 `ThreadPoolExecutor` 并行执行。

### 理由
- 多个检索工具之间通常无依赖，并行化可显著减少总响应时间。
- Python 的多线程在 I/O 密集型任务（网络请求、磁盘读取）中表现良好。

### 后果
- 需注意线程安全，当前各检索器无共享状态，安全可行。

---

## 11. 多轮对话记忆

### 决策
在 AgentState 中维护 chat_history（最近 3 轮对话），前端发送请求时携带历史，后端在规划和生成 Prompt 中注入历史。

### 理由
- 支持指代消解和上下文连贯的多轮对话，提升用户体验。
- 通过限制历史长度（最近 6 条消息），避免 Prompt 过长导致成本增加和性能下降。

### 后果
- 更复杂的 API 模型（需传递 history），但前端封装后对用户透明。

---

## 12. 前端工程化：模块化 SPA + 安全清洗

### 决策
采用原生 HTML/CSS/JS 构建单页应用，拆分为多个 ES 模块（api, store, renderer, utils, main），并引入 DOMPurify 清洗 LLM 输出。

### 理由
- 模块化提高了可维护性和可测试性，展示了专业前端工程能力。
- DOMPurify 防止 LLM 生成恶意脚本，保障安全。
- 无需前端构建工具，降低依赖和入门门槛。

### 后果
- 牺牲了现代框架（React/Vue）的开发效率，但作为演示项目已足够。

---

## 13. 配置管理：pydantic-settings

### 决策
使用 `pydantic-settings` 统一管理所有配置项，支持 `.env` 文件和环境变量。

### 理由
- 类型安全、自动验证，避免配置错误。
- 支持 SecretStr 保护敏感信息（API Key）。
- 所有魔法数字（top_k、chunk_size、retry 次数等）集中管理，便于调整和部署。

### 后果
- 需额外学习 pydantic-settings，但收益明显。

---

## 14. 缓存设计：内存实现 + Redis 接口

### 决策
当前使用 MemoryCache（基于字典），并定义了 BaseCache 抽象类，预留 RedisCache 实现。

### 理由
- 内存缓存实现简单，满足演示需求。
- 通过抽象基类实现了依赖倒置，未来切换至 Redis 只需添加新实现，业务代码无需改动。

### 后果
- 内存缓存重启即丢失，且不可跨进程共享，仅适合单机演示。

---

## 15. 生成答案中移除引用编号

### 决策
修改 Prompt，要求 LLM 直接给出答案，不标注引用编号。

### 理由
- 初期编号标注容易干扰回答格式（如开头多余 [1]），影响阅读体验。
- 当前的 citation_map 已保留在后端，如需展示来源可后续通过前端额外处理。

### 后果
- 削弱了答案的可解释性，但可通过其他方式（如显示检索片段）弥补。

---

## 16. 异步任务与状态轮询

### 决策
文档上传采用 FastAPI BackgroundTasks 异步处理，前端通过轮询 `/task/{task_id}` 获取进度。

### 理由
- 大文件入库耗时长，异步化避免阻塞 API 响应。
- 轮询实现简单，无需引入 WebSocket 或消息队列，降低部署复杂度。

### 后果
- 轮询效率较低，后期可升级为 WebSocket 推送或 Celery 任务队列。

---

## 总结
上述决策相互配合，构成了一个功能完备、工程成熟、可演进的 RAG 智能体系统。每一项技术选型都考虑了当前项目的阶段目标和未来扩展性，确保系统在求职面试中既能展示深度，又不失实用性。
